# Каталог намеренных багов (QA-стенд)

> **Это учебный контракт, не production-ready.**  
> Баги **не чинить** в обычных PR продукта. Чинить только осознанно (режим `strict` / отдельная волна) с обновлением автотестов.

**Режим стенда (E2E):** `KNOWN_BUGS_MODE`  
- `teaching` (по умолчанию) — дефекты живут в коде; тесты с «правильными» ожиданиями помечают `test.fail()`.  
- `strict` — тот же код ожиданий **без** `test.fail()`: CI красный, пока баг не исправлен в продукте. Нужен, чтобы проверить «зелёный стенд после фикса».

В обычном PR CI и локальной работе: **не** задавать `KNOWN_BUGS_MODE=strict`.

## BUG-01 — терминология статуса «ремонт»

| Поле | Значение |
|------|----------|
| **Правильное поведение** | Подпись фильтра и текст в таблице/карточке — **одна** формулировка (например оба «В ремонте»). |
| **Факт стенда** | Фильтр: «Ремонт»; ячейка статуса: «В ремонте». Фильтрация по `repair` работает. |
| **Где в коде** | `BikeFilters.tsx` (подпись), `BikeTable.tsx` `STATUS_LABELS`. |
| **Цель обучения** | Найти inconsistency в copy; не путать «баг UX» с «баг фильтра». |
| **Owner** | Frontend (fix labels) / QA (регрессия). |
| **Автотесты** | `known-bugs.spec.ts` — ожидание равенства текстов + `test.fail` в `teaching`. |

## BUG-02 — ловушка TEST / 123

| Поле | Значение |
|------|----------|
| **Правильное поведение** | Валидные brand=`TEST`, model=`123` → успешный create (**201**), без security-ошибки про guest. |
| **Факт стенда** | POST create → **403** `{ error: "Ошибка безопасности: Ваша роль [guest] …" }` (роль в cookie не guest). UI: `form-server-error`. |
| **Где в коде** | `bikeService.createBike` ловушка; контроллер мапит «роль» → 403. |
| **Цель обучения** | Отличать реальный authZ fail от hardcoded trap; читать текст ошибки критически. |
| **Owner** | Backend / QA. |
| **Автотесты** | UI: `test.fail` на успешный save; API: факт 403 (зелёный) + целевой 201 с `test.fail` в `teaching`. |

## BUG-03 — границы года выпуска

**Целевой диапазон (требования):** 1990 … (текущий календарный год **+ 1**).  
**Текст** верхней ошибки всегда: «Год не может быть позже **{текущий год}**» (не `{текущий+1}`).

| Год | Целевое (требования) | Факт стенда (BUG-03) |
|-----|----------------------|----------------------|
| 1988 | 400 / `error-year` | **201** (проходит) |
| 1989 | 400 | **400** «…раньше 1990» |
| current+1 | 400 | **400** «…позже {current}» |
| current+2 | 400 / `error-year` | **201** (проходит) |

| Поле | Значение |
|------|----------|
| **Цель обучения** | Off-by-one: границы в спецификации vs «срабатывает только на соседних значениях». |
| **Owner** | Fullstack (Zod UI + DTO + service) / QA. |
| **Где в коде** | `frontend/src/schemas.ts`, `backend/src/schemas/dto.ts`, `bikeService.validateYear`. |
| **Автотесты** | UI NEG-06/08 зелёные; NEG-07/09 + `test.fail`; API матрица 4 значений в `known-bugs-api.spec.ts`. |

## Режимы `teaching` / `strict`

| | teaching (default) | strict |
|--|-------------------|--------|
| Код продукта | баги **есть** | ожидается **фикс** багов |
| `markExpectedFailure()` | вызывает `test.fail()` | no-op |
| CI | зелёный (ожидаемые fail) | красный, пока продукт не исправлен |
| Env | unset / `teaching` | `KNOWN_BUGS_MODE=strict` |

Хелпер: `e2e/src/helpers/known-bugs.ts`.

## Чеклист при «починке» бага

1. Убрать ловушку / выровнять labels / заменить year refine на полный диапазон.  
2. Убрать `markExpectedFailure` / `test.fail` для этого ID.  
3. Обновить этот каталог + §7 ТЗ + ручные ТК.  
4. Не смешивать с product-фичами в одном PR без явной пометки PM.
